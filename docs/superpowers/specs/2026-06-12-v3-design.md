# Dinnermate v3 ‚Äî Design

**Date:** 2026-06-12
**Mode:** Delegated ("go ahead and pursue all your other suggestions"). Decisions flagged **[DECISION]** for review. Abuse hardening explicitly excluded per Spencer.

Four features: room-from-list, OSM/Overpass restaurant provider, room durability, PWA polish. Followed by a separate FUTURE.md cycle (meal-time filtering) once this ships.

## Foundation: optional restaurant data **[DECISION]**

OSM has no ratings/price; free-form list items have no coordinates. Rather than fake values, restaurant data becomes honestly optional:

```rust
// Restaurant changes (core):
pub cuisine: Option<String>,
pub price_level: Option<u8>,
pub rating: Option<f32>,        // None = unrated, never 0.0
pub rating_count: Option<u32>,
pub lat: Option<f64>,
pub lng: Option<f64>,
// address stays String (may be empty), name stays required.
```

Migration `0003`: drop NOT NULL on `room_restaurants.{cuisine, price_level, rating, rating_count, address, lat, lng}`.

Filter semantics with unknowns (**fairness rule: unknown passes**): price filter only excludes when `price_level` is `Some` and outside the window; same for rating; radius cut only applies when coords are `Some`; cuisine filter only excludes when `cuisine` is `Some` and not in the selected set. Ordering: rated-above-unrated, then rating desc, then name.

UI: $-signs/‚òÖ hidden when null ("New to us" caption where rating would be), map section hidden without coords, cuisine chip hidden without cuisine (gradient falls back to a neutral warm tone, emoji üçΩÔ∏è). Wire: nullable JSON fields; client models updated.

This ripples through seed tests, fakes, DTOs, and the Dart models ‚Äî contained but wide; the implementation plan sequences it first.

## 1. Room from a list

- `POST /api/v1/rooms/from-list` `{list_code, name?}` ‚Üí 201 `{room, deck}` (member-only ‚Üí 403 `NOT_LIST_MEMBER`; empty list ‚Üí 422 `INVALID_PARAMS` "list has no items").
- Deck = the list's items mapped to restaurant snapshots: `restaurant_id = source_restaurant_id ?? "list-" + item.id`, fields copied (all optional now), position = list order. No provider call, no filtering ‚Äî the list is already curated.
- **[DECISION]** `rooms` gains `source_list_name TEXT NULL` (display only: "From list: Date night" on the room screen). Room params are synthesized for storage (`location_label` = list name, lat/lng/radius from defaults, cuisines empty) and ignored ‚Äî search params are meaningless for a curated deck. No `RoomParams` schema change (YAGNI; a `source` enum can come later if list-rooms diverge further).
- Details endpoint: ids starting `list-` skip the provider and return snapshot + empty extras (no error noise).
- Client: list detail screen gains a primary **"Swipe this list"** button (members only) ‚Üí create ‚Üí straight to the share/join flow reusing the create-room success screen. Room screen shows the "From list" tag.

## 2. OSM/Overpass provider **[DECISION: becomes the production default]**

`OsmProvider` in the api crate (I/O lives next to `google.rs`):

- POST `{OVERPASS_URL}/api/interpreter` (default `https://overpass-api.de`, env-overridable; `User-Agent: dinnermate/1.0 (dinnermate.coolify.stolworthy.co)` per Overpass usage policy), body `data=` Overpass QL:
  `[out:json][timeout:10]; nwr["amenity"="restaurant"]["name"](around:RADIUS,LAT,LNG); out center 60;`
- Mapping: `id` = `osm-<type>-<id>`; `name` (entries without a name are skipped by the query); `cuisine` tag ‚Üí first `;`-token, lowercased, `_`‚Üíspace; address from `addr:housenumber + addr:street + addr:city` (else empty); `lat/lng` from node coords or way/relation `center`; rating/price/rating_count ‚Üí `None`; photo ‚Üí `None`.
- `opening_hours` tag ‚Üí simple-subset parser in **core** (`osm_hours.rs`, pure function, reusable): handles `24/7`, `Day-Day HH:MM-HH:MM` lists with `;`-separated rules and `,`-separated time ranges (e.g. `Mo-Fr 11:00-14:30,17:00-22:00; Sa-Su 12:00-23:00`), day abbreviations Mo/Tu/We/Th/Fr/Sa/Su. Anything else (months, "off", sunrise, PH) ‚Üí `None`. `utc_offset_minutes` ‚Üí `None` (no tz lookup; FUTURE.md gets a note re `tzf-rs`) ‚Äî so OSM cards show weekly hours but no live open/closed badge.
- `details()` ‚Üí empty `ProviderDetails` with `maps_url = Some("https://www.openstreetmap.org/<type>/<id>")`.
- Errors: non-2xx/timeout ‚Üí `Unavailable`; parse failure ‚Üí `InvalidResponse`. Zero results after mapping ‚Üí the existing "no restaurants match" 422 path.
- Config: `RESTAURANT_PROVIDER=osm`. Tests stub Overpass with canned JSON (same pattern as google.rs). **Live verification against real Overpass is possible (no key) and required before flipping production.** After prod verify: set Coolify env `RESTAURANT_PROVIDER=osm` (seed remains the dev/test default).

## 3. Room durability

- **Recent rooms (client-only):** `RecentRooms` service over the existing injectable `KeyValueStore` ‚Äî `[{code, label, last_seen}]`, capped at 5, updated on successful room load/join. Home screen gains a "Jump back in" section (compact chips: label + code) above the lists entry. Removing happens implicitly: tapping an expired room ‚Üí 404 ‚Üí friendly "This room has ended" screen + the entry is dropped.
- **Participants:** `GET /rooms/{code}` response gains `participants: [{display_name}]` (ordered by joined_at). Room screen shows a compact "Alice, Bob +2" line under the room code; matches screen keeps its count.
- **Expiry [DECISION: 30 days]:** tokio background task in api `main.rs` (spawned interval, every 6h): `DELETE FROM rooms WHERE created_at < now() - interval '30 days'` (cascades to deck/participants/swipes). Logged at info with deleted count. Lists never expire.

## 4. PWA polish

- **[DECISION]** `client/web/` becomes checked-in (remove from `.gitignore`): customized `manifest.json` (name "Dinnermate", short_name "Dinnermate", `#FF7E6B` theme, `#FAF6F1` background, standalone, portrait), `index.html` (title, description meta, theme-color, apple-touch-icon + iOS meta), icons 192/512 + maskable variants + favicon ‚Äî generated from a hand-authored SVG (coral rounded square, white fork/spoon/heart mark) via local `rsvg-convert`/ImageMagick, committed as PNGs.
- Client `Dockerfile` drops the `flutter create . --platforms=web` step (web/ is now source); base flutter create artifacts (`flutter_bootstrap.js` reference etc.) match the current stable template.
- Verification: Lighthouse-style manual checks ‚Äî manifest reachable, icons load, installability criteria (served HTTPS in prod, manifest + icons + service worker ‚Äî Flutter web ships `flutter_service_worker.js` already).
